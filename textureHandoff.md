## The corona bug (bisected, one step from fixed)

Symptom: corona meshes exist, enabled, in frustum, material ready — but render
nothing.

Bisect results (headless probe, park corona 4u in front of camera):
- Control plane, plain emissive StandardMaterial → **renders**.
- Corona as-is (emissiveTexture + opacityTexture + ALPHA_ADD) → **invisible**.
- Same corona with `material.opacityTexture = null` → **renders** (white square).
- DynamicTexture canvas is correct (center pixel RGBA = 254,254,254,249).

⇒ The `opacityTexture` path is the killer (Babylon **4.0.3** quirk with
DynamicTexture-as-opacity + ALPHA_ADD). Fix direction: **drop opacityTexture
entirely** — additive blending doesn't need alpha falloff, black adds nothing:

1. Paint the 128px DynamicTexture as an **RGB** radial gradient, white center →
   black edge, on an opaque black fill (don't rely on canvas alpha: canvas
   keeps RGB=white where alpha=0, which additive-adds a white square).
2. `mat.emissiveTexture = dt; mat.disableLighting = true;`
   `mat.alpha = 0.9999` (forces the blend path) `; mat.alphaMode = ALPHA_ADD;`
   `mat.disableDepthWrite = true; mat.backFaceCulling = false`.
3. If ALPHA_ADD still multiplies by src alpha and looks dim, try
   `BABYLON.Engine.ALPHA_ONEONE`.

Verify with the same trick that found it: shot-map probe, or park-in-front
one-liner (see transcript; camera position must be `cam.globalPosition`, NOT
`cam.position` — the rig is parented and local pos is ~origin). Menu overlay
must be hidden before screenshots (`['entry-overlay','splash','menu','main-menu']`).

Good vantages once fixed (world coords, shot-map.mjs already has them):
- lava grotto: (-8.2, 5.2, -10.5) yaw -π/2 — red strobe pool
- torch pit: (0.1, -0.2, 22.3) yaw 0, pitch down — the b=255 TorchWaver

## User asks still open (answer/execute tomorrow)

1. **"instead of repeating the same png have like 20"** — texture variation.
   Modern games solve tiling repetition with: stochastic/hex-grid tiling
   ("texture bombing" — random rotate/offset per tile in shader), texture
   arrays with per-instance indices, macro-variation maps layered at a bigger
   scale, trim sheets + decals, and vertex-painted layer blends; plus lightmaps
   /AO breaking uniformity (our vertex bake already helps here).
   Practical for our Babylon 4 + BSP pipeline, in order of bang/buck:
   a. Generate 3-4 Gemini variants per big texture (script exists!), then
      **split each material's face group into spatial clusters** at load time
      and assign variants per cluster — no shader work, pure mesh split.
   b. A tiny custom shader doing stochastic tiling from a variant atlas
      (riskier on Babylon 4.0.3).
   Recommend (a); the 6 already-generated texture sets are the pilot.
2. **Gallery deploy** — user asked for "a page with all the textures": deploy
   `public/dev/` (see above) and hand them
   https://sol-pkmn.fun/dev/textures.html to pick remaster candidates.
3. **"expected cool lights"** — the bake IS live (red torch hall reads great)
   but the *visible glow* payoff is the corona fix above.
